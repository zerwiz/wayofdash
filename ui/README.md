# Fibonacci Dashboard OS

A React-based information architecture dashboard that implements the Fibonacci Information Architecture Rule ($1 \to 2 \to 3 \to 5 \to 8$). This system allows you to manage documentation, research notes, tasks, and visualizations through an AI-powered chat interface.

## ✨ Features

- **AI-Powered Chat Interface**: Create documents, notes, and tasks through natural conversation
- **Fibonacci Hierarchy**: Navigate information from abstract summaries to technical details using Fibonacci scaling
- **Multi-View System**: Browser, AI Chat, Mindmap, Calendar, and Tasks views
- **Real-time Document Management**: Create and save files directly from chat conversations
- **Interactive Mindmap**: Visual topology map showing project structure
- **Responsive Design**: Works on mobile, tablet, and desktop
- **Host-style dialogs**: File mutations use **`HostMessageModal`** (origin header + Cancel / OK) instead of `window.confirm` / `window.alert`

## 🎓 Fibonacci Architecture

The system follows the Fibonacci Information Architecture Rule:

```
Level 1 (Singularity): 1 node - Executive Summary
Level 2 (Duality):      2 nodes - Primary Pillars
Level 3 (Triad):        3 nodes - Functional Categories
Level 4 (Convergence):  5 nodes - Implementation Details
Level 5 (Depth):        8 nodes - Technical Specifications
```

- Node.js v18+ and npm/yarn installed


```bash

VITE_APP_TITLE=Fibonacci Dashboard
```

## 💻 Usage

### Development Mode

```bash
npm run dev
```

Open your browser to `http://localhost:5173` to see the application.

You’ll land on a **placeholder login** screen first; **Log in** stores a session flag in **`localStorage`** (`way-of-logged-in`) so refreshes stay on the dashboard until you **sign out** (header icon) or clear site data. Replace this with real authentication when you add users.

With the dev server running, the left file tree lists everything under **`workspace/`** (editable workspace). Project documentation uses the folder name **`docs`** inside that tree (e.g. **`workspace/docs/README.md`**). Markdown files open in the main panel with **GitHub-like layout and spacing** via [github-markdown-css](https://github.com/sindresorhus/github-markdown-css) (scoped per theme in `src/styles/gh-md-*.css`, generated by `npm run scope:markdown-css`). Behavior and constraints are documented in **[rules/markdown-rendering-rules.md](../rules/markdown-rendering-rules.md)**.

**Moving items:** Drag the **grip** (⋮⋮) onto a folder or **Workspace** root. On typical phones/tablets (coarse pointer), use the small **folder** icon on a row to enter move mode, then tap the destination folder. **Rename:** use the **pencil** on a row, edit the name, press **Enter** or blur to save, **Escape** to cancel. Renames call `POST /api/files-mutate` with `action: "rename"`. If `/api/data-tree` is unavailable (for example static hosting with no backend, or opening built files without a server), the UI stays in **preview mode** (amber notice): drag-and-drop and live `workspace/` edits are disabled. Use **`npm run dev`** from this folder and open the local URL, or **`npm run build`** then **`npm run preview`** locally — both run the same Vite data API as development.

### Production Build

```bash
npm run build
npm run preview
```

After **`npm run preview`**, open the printed local URL (default `http://localhost:4173`). That server includes the same `/api/*` routes as **`npm run dev`**, so file-tree behavior matches dev. Deploying only the **`dist/`** folder to a static host (for example Netlify) does **not** include those APIs.

### Available Scripts

- `npm run dev` - Start development server
- `npm run build` - Create production build
- `npm run preview` - Preview production build locally (includes workspace/agents API, same as dev)

## 📂 Folder Rules

This project follows the documented folder structure rules:

### User document folders (under `workspace/`)

Paths are relative to **`workspace/`** beside `ui/` (see [folder structure rules](../../rules/folder-structure-rules.md)).

| Folder | Purpose | Access |
|--------|---------|--------|
| `docs/` | Core documentation (Markdown; rendered like GitHub in the UI) | Read/write in dev via `workspace/` APIs |
| `notes/` | Personal notes and research | Read/write in dev |
| `tasks/` | Task lists (e.g. `tasks/task-lists.json` in dev) | Read/write in dev |
| `mindmap/` | Visual diagrams and mindmaps (`.mmd`) | Read/write in dev |
| `ui/` | UI app (this directory) | Developer |

### AI Chat Capabilities

The AI chat interface can:

1. **Create Documents**: Ask "Create a spec for the API" to generate new markdown files
2. **Take Notes**: Say "Take notes on market analysis" to create notes
3. **Add Tasks**: Command "Add a task for updating docs" to create TODOs
4. **Generate Diagrams**: Request "Create a mindmap of the project" for visualizations
5. **Save System Content**: AI-generated content is automatically saved to appropriate folders

### AI Chat Commands

```
"Create docs about {topic}" → Creates document in docs/
"Take notes on {subject}"   → Creates note in notes/
"Add a task {task}"         → Creates task in tasks/
"Make a diagram {desc}"     → Creates `.mmd` in mindmap/
"Save this"                 → Saves current chat output
```

## 🌐 Deployment

### Build for Production

```bash
npm run build
```

This creates a `dist/` folder with optimized static files.

### Deploy to Static Host

- **GitHub Pages**: Deploy from `dist/` folder
- **Netlify**: See [Netlify](#netlify) below
- **Vercel**: Connect your repository
- **AWS S3**: Upload artifacts to S3 bucket

### Netlify

The repo includes **`netlify.toml`** at the project root (parent of `ui/`). It sets `build.base = "ui"` so Netlify installs and builds inside `ui/` and publishes `ui/dist`.

1. In [Netlify](https://app.netlify.com/), choose **Add new site → Import an existing project** and connect your Git provider.
2. Pick the repository whose root contains both `ui/` and `netlify.toml`.
3. Netlify should auto-detect: **Build command** `npm ci && npm run build`, **Publish directory** `dist`, with **Base directory** `ui` (from `netlify.toml`). If anything differs, set **Base directory** to `ui`, **Publish directory** to `dist`.
4. Deploy. SPA fallback is handled by `netlify.toml` and by `public/_redirects` copied into `dist`.

**Static hosting limits:** Production on Netlify is **static files only**. There is no Vite dev server, so **live** features that call `/api/data-tree`, `/api/data-content`, `/api/files-mutate`, `/api/tasks-data`, etc. **do not run** unless you add [Netlify Functions](https://docs.netlify.com/functions/overview/) or another backend. The UI still loads; browsing **`workspace/`** (including **`workspace/docs/`**) and AI file APIs will only work in **`npm run dev`** locally (or if you later wire serverless endpoints).

### CI/CD Pipeline Example

```yaml
# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v3
        with:
          name: dist
          path: dist/
```
